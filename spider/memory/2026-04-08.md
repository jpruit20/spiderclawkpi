## Decisions
- 2026-04-08 - Division Operating Contract enforced system-wide in frontend contract layer - all required KPI surfaces must converge on `KPIObject`, `ActionObject`, `truth_state`, and blocked-state handling rather than ad-hoc page logic.
- 2026-04-08 - AWS telemetry first live source chosen as DynamoDB `sg_device_shadows` - use `device_id` as canonical device key, `sample_time` as sample timestamp, and `device_data.reported.engaged` as active-cook/session derivation signal.
- 2026-04-08 - Sensitive shadow fields must not enter KPI/UI truth path - do not ingest/surface `alertToken`; do not use `ssid` in KPI/UI surfaces.
- 2026-04-08 - Telemetry session derivation remains explicitly proxy-based for now - disconnect/staleness inferred from sample gaps and cook success inferred heuristically until canonical event truth exists.
- 2026-04-08 - After every major KPI site change, push to GitHub and redeploy the live site so production reflects the current repo state rather than waiting on stale Vercel deployments.

## Open follow-up
- Current DynamoDB scan strategy is working but not production-grade for global recent-time telemetry; it needs a better access path (time-oriented export, GSI, or stream/warehouse feed).
- Test-device exclusion is improved with config knobs but still needs an explicit production denylist / metadata-driven rule set.
- Telemetry-specific data-quality validation is still thin and should surface scan completeness, timestamp skew, duplicate samples, and inferred-vs-actual confidence warnings.
